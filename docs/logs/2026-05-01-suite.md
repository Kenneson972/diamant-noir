# Session 2026-05-01 (suite) — Correctifs DB, migration profiles, debug login

## Contexte
Après la création des nouveaux dashboards proprio (10 pages) et admin (7 pages) dans des route groups Next.js 15 distincts, impossibilité de se connecter. Le login redirigeait vers `/login` en boucle, avec une erreur 404 dans la console navigateur.

## Actions réalisées

### 1. Diagnostic base de données
- Constat : la table `public.profiles` n'existe pas dans Supabase (absente des migrations historiques)
- Création de `scripts/check-owner-id.mjs` : script Node.js pour diagnostiquer `owner_id` et profiles
- Résultat : 0 profiles, villas sans `owner_id`

### 2. Migration SQL — Création de la table profiles
**Fichier** : `supabase/migrations/20260501_create_profiles.sql`
- Table `public.profiles` avec colonnes : `id` (uuid FK auth.users), `email`, `full_name`, `phone`, `role` (text avec check), `avatar_url`, `created_at`
- RLS activée : SELECT pour l'utilisateur lui-même, INSERT/UPDATE pour l'utilisateur (id = auth.uid()), admins only for all
- Trigger `on_auth_user_created` : INSERT automatique dans profiles lors de la création d'un utilisateur auth
- Exécutée via SQL Editor Supabase

### 3. Scripts de seeding
**Fichier** : `scripts/seed-test-owners.mjs`
- Création de comptes test via Supabase Admin API
- Testé mais non finalisé (abandonné au profit de la migration)

### 4. Correctifs auth/redirect
- `app/login/page.tsx` : redirect par défaut changé de `/dashboard/proprio` → `/dashboard`
- `middleware.ts` : ajout de la règle de redirect `/dashboard/proprio/*` → `/dashboard/*`
- Build complet OK

### 5. Problème persistant — Session non reconnue
**Symptôme** :
1. Login sur `/login` → OK (utilisateur authentifié)
2. Redirect vers `/dashboard` → middleware intercepte
3. Middleware appelle `supabase.auth.getUser()` → retourne `null` (session non trouvée)
4. Redirect vers `/login?redirect=%2Fdashboard`
5. Boucle infinie

**Hypothèses** :
- Cookies de session Supabase non transmis correctement via `@supabase/ssr` dans le middleware
- Le projet Supabase distant pourrait nécessiter que la migration profiles soit exécutée et que les utilisateurs existants aient leur profile créé manuellement
- Présence d'un process zombie sur le port 3000 (process 18171) forçant le dev server sur 3003 — possible confusion navigateur

**État actuel** :
- Build : OK (0 erreurs TypeScript)
- Dev server : tourne sur `http://localhost:3003`
- Login → redirect → middleware ne détecte pas la session → `/login`
- À résoudre : diagnostiquer la lecture des cookies Supabase dans le middleware Edge
